# Analyze.ps1 レビュー メモ

## 総評

Analyze.ps1 はかなり良い。端的に言えば、もう「補助スクリプト」ではなく、toolkit の中核として成立している。

しかも、ただ機能が多いのではなく、
- 検知
- サニタイズ
- 移行ガイド
- HTML
- CSV

が一本の流れにまとまっている。ここは明確に強い。

## 良い点

### 1. レビュー体験まで設計している

単に危険パターンを数えるだけでなく、
- どの行か
- 何のカテゴリか
- 代替は何か
- サニタイズしたか

まで一画面で追える。

この「検知結果を読む体験」まで面倒を見ているのはかなり良い。紅茶で言えば、茶葉の成分表だけでなく、抽出ログと香りのメモまで同じトレイに載せている。

### 2. HTML がかなり良い

- サイドバー
- コードビュー
- アウトライン
- ツールチップ
- コピー可能な代替例

このあたりが揃っていて、実務レビューにかなり効く。

PowerShell 製の社内ツールでここまで視認性に寄せているのは結構ちゃんとしている。しかも「凝った UI を作りたい」ではなく、「レビューで迷子にさせない」方向に振っているのが良い。

### 3. サニタイズの考え方が実務的

- 元ファイル非破壊
- コピー側だけコメントアウト
- Declare 行だけでなく API 呼び出し側も追う

ここはかなり良い。雑に Declare だけ消して呼び出しが残る、という事故を避けにいっている。現場を見ている。

### 4. テキストレポートと CSV も自然

- EDR
- 互換性
- COM 使用
- API 使用
- 外部参照
- Summary

と、読む順番が自然。

CSV も最低限の棚卸しには十分機能している。

### 5. 設計思想そのものが良い

特に高く見たい点:
- 非破壊
- レビュー中心
- 代替案提示
- 検知だけで終わらない

ただ危険箇所を赤くするツールではなく、「どう扱うか」まで一歩踏み込んでいる。ここは本当に良い。

## 気になる点

### 1. Analyze.ps1 が太り始めている

現在の Analyze.ps1 は、
- 検知
- サニタイズ
- 表示用再構成
- テキストレポート生成
- HTML 生成
- ツールチップデータ生成
- CSV 出力

まで全部抱えている。

今はまだ読み切れるが、今後
- env / biz / info
- 判定列
- 支社向け簡易出力

まで入れ始めるとかなり危ない。急須一つで茶葉の選別から湯温管理、抽出、配膳までやっている状態。

### 2. 内部構造を分けたほうがよい

表向きは Analyze 一本でよいが、中は少なくとも以下くらいに分けたい。

- `Invoke-SanitizePass`
- `Build-AnalyzeTextReport`
- `Build-AnalyzeHtml`
- `Build-AnalyzeCsvRow`

今のままでも動くが、今後の拡張コストがかなり変わる。

### 3. highlight 構築が現在の3カテゴリ前提に寄っている

今は
- `hl-sanitized`
- `hl-edr`
- `hl-compat`

で綺麗に回っているが、これに env / biz / info が入ると、
- 優先順位
- 色クラス
- アウトライン色
- ツールチップ対象

を全部増やす必要がある。

今の分岐直書き方式だとかなり散らばる。

対処案:
各ハイライトを以下のような構造で持つ。
- `Color`
- `Category`
- `PatternName`
- `Priority`
- `TooltipKey`

表示ロジックをデータ駆動に寄せる。

### 4. ツールチップ生成が少し危うい

今は
- API 呼び出し名
- API 宣言名
- パターン名

をその場で JS 文字列にして積んでいる。

今は持っているが、カテゴリが増えると
- 重複
- 命名揺れ
- エスケープ漏れ

で濁る可能性がある。

対処案:
- 一度 hashtable でユニーク化してから JS 化する
- 「見せるためのデータ」を先に整えてから HTML に流す

今は少し直結しすぎている。

### 5. CSV が次の拡張で痛くなる

今はヘッダも行も手で並べているが、今後は以下が入る。

- EnvIssues
- BizIssues
- InfoCount
- RiskLevel
- MigrationClass
- PrimaryConcern
- NeedsReviewBy
- TopApiNames
- TopComProgIds
- SampleEvidence

このあたりを足すと、かなり編集点が増える。

対処案:
- `ordered hashtable` で row を作る
- 列順も定義から出す

今の規模ならまだ間に合う。

### 6. コードの塊の分かれ方が、思想の整理に追いついていない

Analyze は「やっていること」は整理されている。

- 検知して
- 必要なら止めて
- 読める形にして
- 次の修正へつなぐ

この流れはかなり美しい。

ただ、実装はまだその美しさを一つの長い手続きの中に押し込めている。

だから今は味はかなりいいが、抽出器の中で少し圧が上がっている。

### 7. 細かい命名と時間の扱い

#### 7-1. `dummyAnalysis.Patterns.Keys` のような名前

役割が分かりにくい。こういう小さな命名が、コード全体の温度を少し下げる。

#### 7-2. `fileSw = $sw.Elapsed.TotalSeconds`

ファイル単位時間ではなく累積時間になっている。

小さいが、直したほうが気持ちがよい。

## ひとことで言うと

- Analyze はかなり強い
- 今の toolkit の中心として十分成立している
- ただし、次の拡張前に内部構造を少し分けたほうがいい

Analyze は味が出ています。かなり良いです。
でも今は、急須の中で全部をやりすぎて少し圧が上がっている。そこを分ければ、かなり長く戦える Analyze になります。
