# env-patterns / probe-macro 仕様レビュー メモ

## 総評

両方とも成功寄り。かなり良い。

- `env-patterns-spec.md` は実務に効く軸が立っている
- `probe-macro-spec.md` もかなり良いが、初版としては少し広い

特に良い点は以下の4つ。

- 環境依存を独立カテゴリにしたこと
- 判定列を仕様化したこと
- 調査マクロを「環境の別冊」に徹したこと
- 優先実装順を書いたこと

## env-patterns-spec.md の評価

方向はかなり正しい。特に、環境依存と業務依存を EDR / 互換性の横に独立カテゴリとして立てたのは良い。SharePoint / OneDrive 移行や支社対応を考えると、以下を検知対象に入れたのは強い。

- 固定ドライブレター
- UNC パス
- ThisWorkbook.Path
- Workbooks.Open

静かに死ぬものをちゃんと拾いに行けている。

### 気になる点

#### 1. RiskLevel と MigrationClass の意味が競合しやすい

例:
- 固定パス多数 → `RiskLevel = Medium`
- ただし `MigrationClass = 要保存先見直し`

この組み合わせ自体はありだが、読む側には「中リスクなのに移行クラスは重い」と見えることがある。

対処:
- `RiskLevel` は技術危険度
- `MigrationClass` は対応方針

この違いを仕様書に一行明記したほうがよい。同じ温度計ではない、と分ける。

#### 2. PrimaryConcern の「最頻カテゴリ」は危ない

件数が多いカテゴリが主問題とは限らない。

例:
- 固定パスが10件
- FindWindow が1件

この場合、実務上の主問題は GUI 操作系 API のほう。

対処:
- 最頻ではなく重み付きにする
- 少なくとも以下は件数より優先して主問題に倒す
  - GUI操作系 API
  - Shell / PowerShell
  - StorageMigration

#### 3. NeedsReviewBy の BusinessOwner 条件が少し広い

Outlook、Word、印刷を全部 `BusinessOwner` に寄せるのは分かるが、実際には `Infra` や `ClientPC` 側の確認も混ざる。

対処:
- 複数付与前提にする
- 単一担当に落とし込まない

#### 4. 固定端末名/IP が実質 IP しか拾っていない

ホスト名の誤検知が難しいのは理解できるが、将来的には「固定接続先」を IP 以外にも広げたい。

候補:
- URL 内ホスト名
- DB 接続先ホスト名
- UNC 以外の接続先文字列

#### 5. ThisWorkbook.Path の Note が少し強い

「URL 形式になる場合がある」と断定すると、開き方や同期状態の差を吸収しづらい。

対処:
- 「SharePoint / OneDrive 環境では期待したローカルパスにならない可能性がある」程度に抑える

## probe-macro-spec.md の評価

かなり良い。特に以下を最初に置いたのは正しい。

- 対象ブックを直接操作しない
- 副作用なし
- CSV 縦持ち
- 設定シート対応

仕様の芯がぶれていない。

また、以下も良い。

- 端末情報、参照、COM、ファイル、印刷、VBIDE が独立している
- 優先実装順が明記されている
- 単一 xlsm に収める方針が支社配布に向いている

### 気になる点

#### 1. 初版としては少し盛りすぎ

仕様としては良いが、初版実装としては広い。今の案件前提なら、以下は後ろへ送ってよい。

- 通信
- 外部実行
- Office 連携の一部

特に HTTP やメールが実態として薄いなら、本文前半で大きく扱うと焦点が散る。

対処:
- 後回し項目は後半に「任意テスト」として畳む

#### 2. DesktopAppCheck の判定方法が弱い

`Application.OperatingSystem` で「ブラウザ起点でないか」を見るのは無理がある。ここは判定不能に近い。

対処:
以下の観測値を出して、人が判断する形に落とす。

- `ThisWorkbook.Path`
- `ThisWorkbook.FullName`
- `Application.PathSeparator`
- 一時保存っぽいパスか
- OneDrive 同期フォルダ配下か

#### 3. PrinterList で WMI を使う案は少し温度差がある

調査マクロは「新環境の基盤能力チェック」なのに、ここだけ WMI 前提に寄ると、WMI が塞がれているだけで印刷調査が濁る。

対処:
- `ActivePrinter` 取得を優先
- PDF 出力を優先
- プリンタ一覧は補助扱いにする

#### 4. 外部実行テストは支社向けに見た目が強い

- PowerShell
- CMD
- WScript

技術的には理解できるが、支社向け調査マクロに入れると警戒されやすい。

対処:
- 案件実態に薄いなら初版では切る
- 入れるとしても任意テスト扱いにする

#### 5. VBIDEAccess は必須扱いにしないほうがよい

技術的には有用だが、Trust Access は組織設定に触れる話なので、支社向け説明では必須にしないほうが配りやすい。

対処:
- 補助観測に落とす
- 必須カテゴリから外す

#### 6. Build-Probe.ps1 で xlsm 生成は少し重い

構想としては面白いが、初版で build まで抱えると toolkit 側の責務がまた太る。

対処:
- 初版は「手動インポートでも可」を主にする
- build は後続フェーズに回す

## 次にやるとよいこと

- `PrimaryConcern` を最頻ではなく重み付きにする
- `RiskLevel` と `MigrationClass` の意味の違いを明記する
- probe は初版対象をもう少し絞る
- `DesktopAppCheck` の判定方法を観測ベースに落とす

## ひとことで言うと

- env-patterns は今の案件にかなり刺さっている
- probe は少し大きいが、芯は合っている

今は湯量が少し多いだけで、茶葉の選び方は間違っていない
